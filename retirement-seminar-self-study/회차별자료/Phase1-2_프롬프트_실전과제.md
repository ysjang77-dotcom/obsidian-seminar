## 1. 프롬프트 골든 공식 이해하기

### 1.1 프롬프트란?

프롬프트(Prompt)는 AI에게 전달하는 **지시문**입니다. 좋은 프롬프트는 AI로부터 원하는 결과를 정확하게 얻을 수 있게 해줍니다.

### 1.2 골든 공식: 역할 + 맥락 + 지시 + 형식

효과적인 프롬프트는 다음 4가지 요소를 포함합니다:

```
┌─────────────────────────────────────────────────────────────┐
│  프롬프트 골든 공식                                          │
├─────────────────────────────────────────────────────────────┤
│  1. 역할 (Role)    : AI가 수행할 역할/전문가 정의            │
│  2. 맥락 (Context) : 배경 정보, 상황, 제약 조건              │
│  3. 지시 (Instruction) : 수행할 구체적인 작업               │
│  4. 형식 (Format)  : 원하는 출력 형태                        │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Good vs Bad 프롬프트 비교

#### ❌ Bad 프롬프트
```
회의록 써줘
```
**문제점**: 어떤 회의인지, 어떤 형식인지, 무슨 내용인지 알 수 없음

#### ✅ Good 프롬프트
```
[역할]
당신은 10년 경력의 기술 문서 작성 전문가입니다.

[맥락]
2024년 1월 15일 오후 2시에 진행된 AI 프로젝트 킥오프 회의입니다.
참석자: 김팀장, 이연구원, 박연구원
주요 안건: 프로젝트 목표 설정, 역할 분담, 일정 수립

[지시]
위 회의 내용을 바탕으로 공식 회의록을 작성해주세요.
다음 내용을 반드시 포함해주세요:
- 회의 개요 (일시, 장소, 참석자)
- 논의 사항 요약
- 결정 사항
- 향후 일정 및 담당자

[형식]
- 마크다운 형식
- 표를 활용하여 역할 분담 정리
- 전체 분량: A4 1페이지 내외
```

## 2. 과제 수행 절차

### Step 1: 업무 문서 주제 선정

본인 업무에서 다음 중 하나를 선택하세요:

| 문서 유형 | 예시 |
|----------|------|
| 회의록 | 주간 회의, 프로젝트 회의, 브레인스토밍 회의 |
| 보고서 | 연구 진행 보고서, 출장 보고서, 분석 보고서 |
| 기획서 | 프로젝트 기획서, 행사 기획서, 제안서 |
| 메일 초안 | 협조 요청, 결과 공유, 문의 응답 |
| 업무 매뉴얼 | 절차 안내, 시스템 사용법, 체크리스트 |

### Step 2: 프롬프트 작성 (최초 버전)

골든 공식을 적용하여 프롬프트를 작성하세요.

#### 프롬프트 템플릿

```markdown
## 프롬프트 v1

[역할]
당신은 _______________ 전문가입니다.

[맥락]
- 상황: _______________
- 대상: _______________
- 목적: _______________
- 제약조건: _______________

[지시]
다음 작업을 수행해주세요:
1. _______________
2. _______________
3. _______________

[형식]
- 출력 형식: _______________
- 분량: _______________
- 톤앤매너: _______________
```

### Step 3: Claude에서 실행 및 결과 확인

1. Claude.ai 또는 Claude Desktop 실행
2. 작성한 프롬프트 입력
3. 결과물 검토

### Step 4: 프롬프트 개선 (반복 2회 이상)

결과물이 만족스럽지 않다면 프롬프트를 수정하세요.

#### 개선 체크리스트
- [ ] 역할이 충분히 구체적인가?
- [ ] 맥락 정보가 누락되지 않았는가?
- [ ] 지시가 명확하고 단계별로 나뉘어 있는가?
- [ ] 원하는 형식을 명시했는가?
- [ ] 예시를 추가하면 더 나아질까?

#### 프롬프트 개선 예시

**v1 → v2 개선**: "공식적인 어조로" 추가
**v2 → v3 개선**: 구체적인 예시 1개 제공

## 3. Hallucination 검증

### 3.1 Hallucination이란?

AI가 사실이 아닌 내용을 그럴듯하게 생성하는 현상입니다. **AI의 답변을 100% 신뢰하면 안 됩니다.**

### 3.2 검증 방법

AI 답변 중 **최소 3건**의 사실 정보를 교차 검증하세요.

#### 검증 대상 예시
- 날짜, 숫자, 통계
- 인용된 법률, 규정
- 인물, 기관 정보
- 기술 용어 정의
- 참고 문헌, URL

#### 검증 기록 양식

| 항목 | AI 답변 내용 | 검증 결과 | 검증 출처 |
|------|-------------|----------|----------|
| 1 | (AI가 말한 내용) | ✅ 정확 / ❌ 오류 | (확인한 출처) |
| 2 | (AI가 말한 내용) | ✅ 정확 / ❌ 오류 | (확인한 출처) |
| 3 | (AI가 말한 내용) | ✅ 정확 / ❌ 오류 | (확인한 출처) |

### 3.3 검증 예시

**예시 1: 법률 규정 검증**
- AI 답변: "개인정보보호법 제17조에 따르면..."
- 검증: 국가법령정보센터에서 실제 조문 확인
- 결과: ✅ 정확

**예시 2: 통계 수치 검증**
- AI 답변: "2023년 한국의 연구개발비는 약 100조원입니다"
- 검증: 과학기술정보통신부 통계 확인
- 결과: ❌ 실제 약 70조원대로 오류

## 4. 프롬프트 개선 히스토리 작성

### 4.1 히스토리 기록 양식

```markdown
# 프롬프트 개선 히스토리

## 문서 주제: _______________

---

### 버전 1 (최초)

**프롬프트:**
[전체 프롬프트 내용]

**결과 요약:**
[AI 답변의 핵심 내용 2-3문장]

**문제점:**
- [개선이 필요한 부분]

---

### 버전 2 (1차 수정)

**변경 사항:**
- [v1에서 변경한 내용]

**프롬프트:**
[수정된 프롬프트]

**결과 요약:**
[개선된 점]

**추가 문제점:**
- [아직 개선이 필요한 부분]

---

### 버전 3 (최종)

**변경 사항:**
- [v2에서 변경한 내용]

**프롬프트:**
[최종 프롬프트]

**최종 결과:**
[만족스러운 결과물 설명]

---

## 학습 포인트

이번 과제를 통해 배운 점:
1. _______________
2. _______________
3. _______________
```

## 5. 프롬프트 작성 실전 팁

### 5.1 역할 설정 팁

```markdown
# 좋은 역할 설정 예시

✅ "10년 경력의 기술 문서 전문가로서"
✅ "연구기관의 행정 업무 담당자로서"
✅ "과학기술 분야 논문 리뷰어로서"

❌ "똑똑한 AI로서" (너무 모호함)
❌ "전문가로서" (어떤 분야?)
```

### 5.2 맥락 제공 팁

```markdown
# 맥락에 포함할 정보

✅ 대상 독자: "이 문서는 비전문가도 이해할 수 있어야 합니다"
✅ 배경 상황: "현재 프로젝트는 초기 단계입니다"
✅ 제약 조건: "예산은 5000만원 이내입니다"
✅ 참고 자료: "첨부한 회의 메모를 참고하세요"
```

### 5.3 지시 작성 팁

```markdown
# 명확한 지시 작성법

✅ 단계별로 나누기
   1. 먼저 ... 분석해주세요
   2. 그 다음 ... 작성해주세요
   3. 마지막으로 ... 검토해주세요

✅ 구체적인 기준 제시
   - "3가지 이상의 개선안을 제시해주세요"
   - "각 항목은 2문장 이내로 요약해주세요"

❌ 모호한 지시
   - "잘 써줘" (무엇이 '잘'인지?)
   - "자세히 설명해줘" (얼마나 자세히?)
```

### 5.4 형식 지정 팁

```markdown
# 출력 형식 예시

✅ "마크다운 형식으로 작성해주세요"
✅ "표 형식으로 비교해주세요"
✅ "번호 매긴 리스트로 정리해주세요"
✅ "JSON 형식으로 출력해주세요"

# 분량 지정 예시
✅ "A4 1페이지 분량으로"
✅ "500자 내외로"
✅ "5개 항목으로"
```

## ❓ 자주 묻는 질문 (FAQ)

### Q1: 어떤 업무 문서를 작성해야 하나요?
**A**: 본인 업무와 관련된 **실제로 필요한 문서**를 선택하세요. 가상의 문서보다 실제 업무에 활용할 수 있는 문서를 작성하면 학습 효과가 높습니다.

### Q2: 프롬프트를 몇 번이나 수정해야 하나요?
**A**: 최소 2회 이상 수정하여 v1 → v2 → v3(또는 그 이상) 히스토리를 만들어야 합니다. 중요한 것은 **왜 수정했는지**를 기록하는 것입니다.

### Q3: Hallucination 검증에서 오류를 발견하면 어떻게 하나요?
**A**: 오류를 발견하면 그대로 기록하고, 올바른 정보로 수정한 내용도 함께 기록하세요. 오류 발견은 **실패가 아니라 학습**입니다.

### Q4: Claude가 긴 답변을 중간에 끊으면 어떻게 하나요?
**A**: "계속 작성해주세요" 또는 "이어서 작성해주세요"라고 요청하세요. 또는 처음부터 "분량: 500자 이내" 등으로 제한을 두세요.
